# DESIGN

## Модель данных
- `approval_requests` хранит заявку: workspace, sourceType/sourceId, title, description, reviewerUserIds, статус и hash запроса.
- `approval_decisions` хранит аудиторскую запись каждого решения: approve/reject/cancel, кто принял решение и комментарий/причину.

## Границы сервиса
- Сервис работает только с идентификаторами внешних сущностей и не реализует соседние сервисы.
- Данные одного workspace не доступны из другого; доступ контролируется через workspace header и auth stub.

## Обработка повторов
- Перед созданием заявки считается `request_hash` по workspace, payload и actor user.
- При совпадении возвращается уже существующая заявка, чтобы не создавать дубли.

## События и интеграции
- При создании и изменении статуса генерируются безопасные события с минимальным набором данных.
- В событиях и логах не попадают секреты, токены, email и provider payload.

## Известные компромиссы
- Для локального запуска используется SQLite через `aiosqlite`.
- Авторизация реализована как заглушка через HTTP-заголовки для упрощения запуска и тестирования.
